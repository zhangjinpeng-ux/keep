## 1. 整体结构

```yaml
workflow:
  id: my-workflow
  name: 我的工作流
  description: 描述
  disabled: false

  # --- 工作流级配置 ---
  triggers: []
  consts: {}
  owners: []
  services: []
  tags: []
  permissions: []
  strategy: nonparallel_with_retry
  debug: false
  interval: 0
  on-failure: {}

  # --- 步骤（数据获取）---
  steps: []

  # --- 动作（通知/操作）---
  actions: []
```

每个字段对应的代码位置：

| YAML 字段 | parser.py 解析逻辑 | Workflow 构造参数 |
|-----------|-------------------|------------------|
| `id` | `_get_workflow_id()` → `workflow_id` |
| `name` | `workflow.get("name", "Untitled")` | `workflow_name` |
| `description` | `workflow.get("description", "No description")` | `workflow_description` |
| `disabled` | `parse_disabled()` | `workflow_disabled` |
| `owners` | `_parse_owners()` | `workflow_owners` |
| `tags` | `_parse_tags()` | `workflow_tags` |
| `permissions` | `workflow.get("permissions", [])` | `workflow_permissions` |
| `consts` | `workflow.get("consts", {})` | `workflow_consts` |
| `strategy` | `workflow.get("strategy", "nonparallel_with_retry")` | `workflow_strategy` |
| `debug` | `workflow.get("debug", False)` | `workflow_debug` |
| `triggers` | `get_triggers_from_workflow_dict()` | `workflow_triggers` |
| `steps` | `_parse_steps()` | `workflow_steps` |
| `actions` | `_parse_actions()` | `workflow_actions` |
| `on-failure` | `_get_on_failure_action()` | `on_failure` |

## 2. 工作流级字段

### `id`（必填）

工作流唯一标识。存储在数据库中，用于引用。

```yaml
workflow:
  id: my-unique-workflow-id
```

### `name`（可选，默认 "Untitled"）

工作流显示名称。

### `description`（可选，默认 "No description"）

```yaml
description: 获取 MySQL 表大小并发送通知
```

### `disabled`（可选，默认 false）

设为 `true` 禁用工作流，不再触发和执行。

```yaml
disabled: true
```

### `owners` / `tags`

用于标记工作流的所有者和标签，仅作标识用。

```yaml
owners:
  - devops-team
tags:
  - mysql
  - monitoring
```

### `permissions`

限制工作流可被谁执行。值为用户 email 或角色名。

```yaml
permissions:
  - admin
  - user@example.com
```

### `consts`

定义工作流级常量，可在模板中用 `{{ consts.xxx }}` 引用。

```yaml
consts:
  alert_severity: critical
  slack_channel: "#alerts"
```

使用：`{{ consts.slack_channel }}`

### `strategy`（可选，默认 `nonparallel_with_retry`）

控制相同 fingerprint 事件的并发策略（`workflow.py` 第 12-18 行 `WorkflowStrategy`）：

| 值 | 行为 |
|----|------|
| `nonparallel_with_retry` | 相同 fingerprint 正在执行则跳过，下一周期重试（默认） |
| `nonparallel` | 相同 fingerprint 正在执行则直接跳过 |
| `parallel` | 允许并行执行 |

### `debug`（可选，默认 false）

启用后输出更详细的执行日志。

```yaml
debug: true
```

### `on-failure`

工作流整体执行失败时的回调 action，语法与 action 一致。

```yaml
on-failure:
  provider:
    type: console
    with:
      message: "工作流执行失败: {{ steps.get-pods.errors }}"
```

## 3. Triggers（触发器）

### 触发器类型

#### `manual` — 手动触发

```yaml
triggers:
  - type: manual
```

#### `alert` — 告警触发

当有告警事件到达时，通过 CEL 表达式判断是否触发。

```yaml
triggers:
  - type: alert
    cel: source.contains("prometheus") && severity == "critical"
```

CEL 可用字段（来自 AlertDto）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `source` | list[string] | 告警来源（用 `.contains()` 匹配） |
| `severity` | string | critical/error/warning/info |
| `name` | string | 告警名称 |
| `status` | string | firing/resolved |
| `environment` | string | 环境标签 |
| `service` | string | 服务名 |
| `description` | string | 描述 |
| `message` | string | 消息内容 |

高级 CEL 写法：

```yaml
# 多条件组合
cel: (source.contains("datadog") && severity == "critical") || (source.contains("newrelic") && severity == "error")

# 字符串匹配
cel: name.contains("error") || name.contains("failure")

# 状态判断
cel: status == "firing"
```

CEL 预处理机制（`cel_utils.py`）：severity 会被转为数值（critical=4, error=3, warning=2, info=1），支持 `>` `<` 比较。

#### `interval` — 定时触发

```yaml
triggers:
  - type: interval
    value: 5m    # 5 分钟执行一次，支持 m（分钟）/h（小时）/d（天）
```

#### `incident` — 事件触发

```yaml
triggers:
  - type: incident
    events:
      - created
      - updated
```

### 触发器过滤增强

```yaml
triggers:
  - type: alert
    cel: source.contains("cloudwatch")
    only_on_change:
      - severity          # 仅当 severity 变化时触发
    severity_changed: true  # 仅 severity 变化时触发（会在 alert 中注入 severity_changed 和 previous_severity）
```

## 4. Steps（步骤）

Step 用于**读取数据**。调用 `provider.query()`。

### 基本结构

```yaml
steps:
  - name: get-mysql-data
    provider:
      type: mysql
      config: "{{ providers.mysql }}"
      with:
        query: SELECT * FROM information_schema.TABLES
        single_row: false
    vars: {}
    condition: []
    foreach: ""
    if: ""
    continue: true
    on-failure: {}
    throttle: {}
```

### 字段说明

#### `name`（必填）

步骤名称，用于在上下文中引用结果：`{{ steps.get-mysql-data.results }}`

#### `provider`

| 子字段 | 说明 |
|--------|------|
| `type` | provider 类型，如 `mysql`、`postgres`、`aks`、`webhook`、`bash` 等 |
| `config` | 引用已配置的 provider：`"{{ providers.xxx }}"` |
| `with` | 传给 provider 的参数，如 `query`、`command_type`、`url` 等 |

```yaml
provider:
  type: aks
  config: "{{ providers.aks }}"
  with:
    command_type: get_pods
```

#### `single_row`（仅 SQL 类 provider）

```yaml
with:
  query: SELECT * FROM table
  single_row: true    # 只返回第一行（dict/tuple），不返回列表
  as_dict: true       # MySQL 专用，返回 dict 列表而非 tuple 列表
```

| `single_row` | Provider 返回格式 |
|---|---|
| `false`（默认） | `[{row1}, {row2}, ...]` 列表 |
| `true` | `{row1}` 单个结果 |

#### `vars`

定义步骤级变量，可用在条件中。

```yaml
vars:
  threshold: 100
  env: production
```

在上下文中通过 `{{ vars.threshold }}` 引用。

#### `foreach`

对 step 结果进行迭代（step 本身也可以 foreach，用于批量查询）。

```yaml
steps:
  - name: query-per-table
    foreach: "{{ steps.show-tables.results }}"    # 遍历每张表
    provider:
      type: mysql
      with:
        query: "SELECT COUNT(*) FROM {{ foreach.value.0 }}"  # foreach.value 是当前行
```

#### `condition`

见第 6 节「条件控制」。

#### `if`

Python 表达式，为 true 时执行该 step。

```yaml
if: "keep.len({{ steps.other-step.results }}) > 0"
```

#### `continue`（可选，默认 true）

该 step 执行后是否继续执行后续 step/action。

```yaml
continue: false    # 执行完此 step 后停止工作流
```

#### `on-failure` / `retry`

```yaml
on-failure:
  retry:
    count: 3           # 失败后重试 3 次
    interval: 5        # 每次重试间隔 5 秒
```

#### `throttle`

```yaml
throttle:
  type: one_until_resolved   # 同一 fingerprint 的告警在未 resolved 前不再重复执行
  with:
    # 不同 throttle 类型的配置
```

## 5. Actions（动作）

Action 用于**执行操作**。调用 `provider.notify()`。

### 基本结构

```yaml
actions:
  - name: send-slack
    foreach: "{{ steps.get-pods.results }}"
    if: "{{ foreach.value.5 }} > 10"
    provider:
      type: slack
      config: "{{ providers.slack }}"
      with:
        message: "Pod: {{ foreach.value.metadata.name }}"
    enrich_alert:
      - key: ticket_id
        value: results.id
```

### Action 特有字段

#### `enrich_alert`

将执行结果写回告警，实现告警富化。

```yaml
enrich_alert:
  - key: ticket_id
    value: results.issue.key    # 引用 provider 返回结果
    disposable: false            # 是否一次性富化
```

`value` 支持两种引用方式：

- `results.xxx.yyy` — 点号路径
- `results[0]["key"]` — 索引访问

#### `enrich_incident`

与 `enrich_alert` 相同，但用于富化事件。

## 6. 条件控制

### 6.1 `if` 表达式

使用 `asteval` 库执行 Python 表达式，结合 `keep.*` 函数：

```yaml
if: "keep.len({{ steps.query.results }}) > 0"
if: "not '{{ alert.ticket_id }}'"
if: "'{{ alert.severity }}' == 'critical'"
if: "{{ foreach.value.5 }} > 10"
```

引用多个值复合判断：

```yaml
if: "keep.len({{ steps.a.results }}) or keep.len({{ steps.b.results }})"
```

### 6.2 `condition` 块

内置的声明式条件，在 `if` 之前评估，结果注入上下文。

#### threshold（阈值条件）

```yaml
condition:
  - name: cpu-high
    type: threshold
    alias: CPU_ALERT
    value: "{{ steps.query.results.0.cpu_usage }}"    # 实际值
    compare_to: 90                                      # 阈值
    compare_type: gt                                    # gt（大于）/ lt（小于）
    level: critical                                     # 多阈值时标记级别
```

多阈值：

```yaml
condition:
  - name: multi-threshold
    type: threshold
    value: "{{ steps.query.results.0.utilization }}"
    compare_to: "60, 80, 95"       # 多个阈值
    level: "warning, critical, severe"   # 对应级别
```

条件别名会在上下文中注册，可在 `if` 中引用：

```yaml
condition:
  - name: cpu-high
    type: threshold
    alias: CPU_ALERT
    value: ...
    compare_to: 90
if: "{{ CPU_ALERT }}"  # 条件成立时为 True
```

#### assert（断言条件）

```yaml
condition:
  - name: check-error
    type: assert
    assert: "'{{ foreach.value.status }}' == 'Running'"
```

### 6.3 `foreach`

遍历列表，每次迭代将当前元素设为 `{{ foreach.value }}`。

```yaml
foreach: "{{ steps.query-tables.results }}"
```

当 `foreach.value` 是数组时，用数字索引访问：

```yaml
# foreach.value = ["recoveryfile", 0.02, 0.02, 0.03, 0, 69]
message: "表名: {{ foreach.value.0 }}, 行数: {{ foreach.value.5 }}"
```

当 `foreach.value` 是字典时，用键名访问：

```yaml
# foreach.value = {"name": "recoveryfile", "rows": 69}
message: "表名: {{ foreach.value.name }}, 行数: {{ foreach.value.rows }}"
```

多 foreach 组合（用 `&&` 分隔）：

```yaml
foreach: "{{ steps.keys.results }} && {{ steps.values.results }}"
# foreach.value 是两个列表 zip 后的元组
```

## 7. 模板变量

所有可用上下文（`contextmanager.py` `get_full_context()`）：

| 变量 | 来源 | 说明 |
|------|------|------|
| `{{ steps.xxx.results }}` | step 执行结果 | 步骤的输出数据 |
| `{{ steps.xxx.conditions }}` | 条件评估结果 | 条件详情列表 |
| `{{ steps.xxx.vars }}` | `vars` 定义 | 步骤变量 |
| `{{ actions.xxx.results }}` | action 执行结果 | `actions` 是 `steps` 的别名 |
| `{{ foreach.value }}` | 当前迭代值 | foreach 循环的当前元素 |
| `{{ foreach.items }}` | 全部迭代项 | foreach 的完整列表 |
| `{{ event }}` | 触发事件 | 当前 alert/incident 事件 |
| `{{ alert }}` | `event` 的别名 | 同 `{{ event }}` |
| `{{ incident }}` | incident 事件 | 当前 incident |
| `{{ consts.xxx }}` | `consts` 定义 | 工作流常量 |
| `{{ vars.xxx }}` | `vars` 定义 | 当前步骤变量 |
| `{{ secrets.xxx }}` | Secret Manager | 密钥 |
| `{{ providers.xxx }}` | provider 配置 | 已安装的 provider 配置 |
| `{{ inputs.xxx }}` | 工作流输入 | 手动触发时传入的参数 |
| `{{ last_workflow_results }}` | 上次执行结果 | 上一次工作流运行的结果 |
| `{{ last_workflow_run_time }}` | 上次执行时间 | |

### keep.* 函数

可在模板或 `if` 中使用的内置函数（`keep/functions/__init__.py`）：

```yaml
# 集合操作
keep.len({{ list }})          # 长度
keep.first({{ list }})        # 第一个
keep.last({{ list }})         # 最后一个
keep.join({{ list }}, ",")    # 拼接
keep.split("a,b", ",")        # 分割

# 数学运算
keep.add(1, 2)                # 加法
keep.sub(10, 3)               # 减法
keep.mul(3, 4)                # 乘法
keep.div(10, 3)               # 整数除法
keep.mod(10, 3)               # 取模
keep.exp(2, 3)                # 幂运算

# 字符串
keep.uppercase("hello")       # 大写
keep.lowercase("HELLO")       # 小写
keep.capitalize("hello")      # 首字母大写
keep.title("hello world")     # 每个单词首字母大写
keep.strip(" hello ")         # 去空格
keep.replace("a,b", ",", "-") # 替换
keep.slice("hello", 0, 3)     # 截取 "hel"

# JSON
keep.json_loads('{"a":1}')    # 字符串 → dict
keep.json_dumps({{ dict }})   # dict → JSON 字符串

# 时间
keep.utcnow()                 # 当前 UTC 时间
keep.utcnowiso()              # ISO 格式时间字符串
keep.utcnowtimestamp()        # 时间戳
keep.to_utc({{ dt }})         # 转 UTC
keep.to_timestamp({{ dt }})   # 转时间戳
keep.datetime_compare(t1, t2) # 比较时间差（小时）

# 其他
keep.dictget({{ dict }}, "key", "default")  # 安全取字典值
keep.dict_pop({{ dict }}, "key")            # 删除键
keep.dict_filter_by_prefix({{ dict }}, "pre") # 按前缀过滤
keep.encode("hello")          # URL 编码
```

## 8. 完整示例

### 示例 1：MySQL 表空间监控

```yaml
workflow:
  id: mysql-table-monitor
  name: MySQL 表空间监控
  description: 查询 MySQL 表大小，超过阈值时通知
  disabled: false
  triggers:
    - type: interval
      value: 1h
  consts:
    size_threshold: 100    # MB
  steps:
    - name: get-tables
      provider:
        type: mysql
        config: "{{ providers.mysql }}"
        with:
          query: |
            SELECT table_name AS `表名`,
                   ROUND(data_length/1024/1024,2) AS `数据MB`,
                   ROUND(index_length/1024/1024,2) AS `索引MB`,
                   ROUND((data_length+index_length)/1024/1024,2) AS `总占用MB`,
                   ROUND(data_free/1024/1024,2) AS `碎片MB`,
                   table_rows AS `估算行数`
            FROM information_schema.TABLES
            WHERE table_schema='production'
            ORDER BY (data_length+index_length) DESC
  actions:
    - name: notify-large-tables
      foreach: "{{ steps.get-tables.results }}"
      if: "{{ foreach.value.3 }} > {{ consts.size_threshold }}"
      provider:
        type: console
        with:
          message: "⚠️ 表 {{ foreach.value.0 }} 占用 {{ foreach.value.3 }}MB，超过阈值 {{ consts.size_threshold }}MB"
```

### 示例 2：AKS Pod 状态检查

```yaml
workflow:
  id: aks-pod-monitor
  name: AKS Pod 状态
  triggers:
    - type: manual
  steps:
    - name: get-pods
      provider:
        type: aks
        config: "{{ providers.aks }}"
        with:
          command_type: get_pods
  actions:
    - name: echo-failing-pods
      foreach: "{{ steps.get-pods.results }}"
      if: "'{{ foreach.value.status.phase }}' != 'Running'"
      provider:
        type: console
        with:
          message: "❌ Pod {{ foreach.value.metadata.name }} ({{ foreach.value.metadata.namespace }}) 状态: {{ foreach.value.status.phase }}"
```

### 示例 3：CEL 触发 + Slack 通知

```yaml
workflow:
  id: critical-alert-handler
  name: 关键告警处理
  triggers:
    - type: alert
      cel: (source.contains("prometheus") || source.contains("datadog")) && severity == "critical" && environment == "production"
    - type: manual
  actions:
    - name: send-slack
      if: "not '{{ alert.ticket_id }}'"
      provider:
        type: slack
        config: "{{ providers.slack-prod }}"
        with:
          message: |
            🚨 CRITICAL: {{ alert.name }}
            来源: {{ alert.source }}
            服务: {{ alert.service }}
            描述: {{ alert.description }}
          channel: "#oncall"
      enrich_alert:
        - key: ticket_id
          value: results.message
    - name: create-jira
      if: "not '{{ alert.ticket_id }}'"
      provider:
        type: jira
        config: "{{ providers.jira }}"
        with:
          summary: "[CRITICAL] {{ alert.name }}"
          description: "{{ alert.description }}"
          issuetype: "Incident"
          enrich_alert:
            - key: ticket_id
              value: results.issue.key
```
