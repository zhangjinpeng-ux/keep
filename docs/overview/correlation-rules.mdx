title: "手动关联规则"

Keep 关联引擎是一个多功能工具，用于将告警关联和整合为事件或事件候选项。
本指南解释了有效使用规则引擎的核心概念、用法和最佳实践。

## 核心概念
- **规则定义**：Keep 中的规则是一组条件，当满足这些条件时，会创建一个事件或事件候选项。
- **告警属性**：这些是告警的特征或数据点，例如来源、严重程度或告警可能具有的任何属性。
- **条件与逻辑**：规则通过基于告警属性定义条件来构建，使用逻辑运算符（如 AND/OR）组合多个条件。

## 创建关联规则
创建规则涉及定义告警应被归类或操作应被分组的条件。

1. **访问关联引擎**：导航到 Keep 平台中的 Correlation 部分。
2. **定义规则条件**：
 - **命名规则**：分配一个描述性名称以反映其用途。
 - **设置条件**：使用告警属性创建条件。例如，一个规则可以指定严重程度为 'critical' 且来源为 'Prometheus' 的告警应被归类为 'High Priority'。
 - **逻辑分组**：使用逻辑运算符组合条件以形成全面的规则。
 - **手动审批**：创建事件候选项或完整事件。

## 动态事件命名

关联引擎支持基于告警属性的动态事件命名。这使您能够创建更有意义和上下文感知的事件名称，反映实际的告警数据。

### 模板变量

您可以使用 `{{ alert.attribute }}` 语法在事件名称中使用模板变量。这些变量会被替换为告警中的实际值。例如：
- `{{alert.labels.host}}` - 引用告警标签中的主机
- `{{alert.service}}` - 引用告警中的服务名称

### 多条告警时的行为

当一个事件包含多条告警时：

- 所有告警的值会自动用逗号连接
- 重复值会自动去重
- 如果新告警添加了唯一值，事件名称会更新以包含该值

#### 动态命名示例

**模板：** "Service Issue on `{{alert.labels.host}}`"

**第一条告警**
```
{
  ...
  {
    "labels": {
      "host": "host1"
    }
  }
  ...
}
```

**第二条告警**

```
{
  ...
  {
    "labels": {
      "host": "host2"
    }
  }
  ...
}
```

**事件名称**

Service Issue on host1,host2

## 示例
- **基于指标的告警**：构建规则以精确定位与特定指标相关的告警，例如服务器上的高 CPU 使用率。这可以通过将共享通用属性（如 'CPU usage' 标签）的告警分组来实现，确保您快速识别并解决性能问题。
- **功能相关告警**：建立规则以按特定功能或服务创建事件。例如，您可以基于 'service' 或 'URL' 标签启动事件。此方法对于跟踪和管理与应用程序内不同功能或组件相关的告警特别有用。
- **基于团队的告警管理**：实施规则以根据团队职责创建事件。这可能涉及根据特定团队负责的系统或服务进行分组。这种策略确保告警及时定向到适当的团队，提高响应时间和效率。
