<Warning>
  如果提供商未声明指纹字段，则指纹默认为告警名称。
</Warning>

指纹在 Keep 的告警上下文中具有以下几个重要用途：

### 去重

告警指纹用于防止对相同的底层告警重复触发富化/工作流。
当 Keep 接收到告警时，它会根据提供商中声明的配置字段计算指纹。
如果两条告警具有相同的指纹，Keep 认为它们是重复的，并只呈现其中一条。
这有助于减少告警噪音，防止不必要的工作流触发/富化。

### 分组

Keep 使用告警指纹将相关告警分组在一起。
具有相同指纹的告警被视为属于同一组，表明它们是由相同的底层条件或问题触发的。
分组告警使运维人员更容易理解不同告警源之间的关系、问题的根因，并更快地采取适当的行动。

### 静默

告警指纹在第三方工具中用于管理静默/抑制。
静默允许运维人员临时抑制具有特定指纹的告警，提供一种在不生成额外通知/触发的情况下确认和处理已知问题的方法。

### 可视化

告警指纹也可用于可视化和分析目的。
它们有助于跟踪告警随时间变化的历史和状态，并提供将告警与受监控系统中特定条件或更改关联起来的方法。

生成指纹的过程涉及对提供商中配置的字段及其与告警实例关联的值进行哈希处理。
这会生成一个固定长度的十六进制字符串，唯一标识该告警。
当 Keep 接收/获取告警时，它会为每条告警计算指纹，以确定是否应触发工作流、进行分组或被静默。

总之，Keep 告警指纹对于在每个第三方系统中管理和组织告警至关重要。
它们有助于防止重复、分组相关告警、启用静默，并促进告警数据的分析和可视化，最终有助于受监控系统的有效运行和维护。

### 示例

这是指纹字段的基础提供商类实现：

```python base_provider.py
class BaseProvider(metaclass=abc.ABCMeta):
    OAUTH2_URL = None
    PROVIDER_SCOPES: list[ProviderScope] = []
    PROVIDER_METHODS: list[ProviderMethod] = []
    FINGERPRINT_FIELDS: list[str] = []
```

这是 Datadog 的指纹字段提供商实现示例，我们基于事件组和监控 ID 计算指纹：

```python datadog_provider.py
class DatadogProvider(BaseProvider):
    """
    Datadog provider class.
    """

    PROVIDER_SCOPES = [
      ...
    ]
    PROVIDER_METHODS = [
      ...
    ]
    FINGERPRINT_FIELDS = ["groups", "monitor_id"]
```
