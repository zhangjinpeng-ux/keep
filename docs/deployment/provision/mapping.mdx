---
title: "Mapping Rule Provisioning"
---

<Tip>对于任何与映射规则预置相关的问题，请加入我们的 [Slack](https://slack.keephq.dev) 社区。</Tip>

Keep 中的映射规则预置允许您在版本控制中而不是在 UI 中管理 CSV 风格的告警增强规则。当您想要将映射更改与基础设施即代码的其他部分一起追踪时，这非常有用。

### 配置映射规则

要预置映射规则，请按照以下步骤操作：

1. 将 `KEEP_MAPPINGS_DIRECTORY` 环境变量设置为映射配置目录的路径。
2. 在该目录中为每个映射规则创建一个 YAML 清单文件。

目录结构示例：
```
/path/to/mappings/
├── prometheus-by-namespace.yaml
├── cloudwatch-by-team.yaml
└── service-topology.yaml
```

### 清单格式

每个 YAML 清单描述一个映射规则。字段与 REST `POST /mapping` 端点接受的字段对应：

```yaml
name: example-prometheus-mapping
description: optional description
priority: 0
type: csv
matchers:
  - [namespace]
rows:
  - { namespace: monitoring, team: platform }
  - { namespace: default, team: platform }
```

| 字段 | 是否必需 | 备注 |
| --- | --- | --- |
| `name` | 是 | 查找键 — 在租户内必须唯一 |
| `description` | 否 | 可读的描述 |
| `priority` | 否 | 整数，默认 `0`。值越高 = 越先评估 |
| `type` | 否 | `csv`（默认）或 `topology` |
| `matchers` | 是 | 属性组列表。在列表内：AND。列表之间：OR |
| `rows` | 是（对于 `csv`） | `{key: value}` 字典列表；行与传入的告警属性匹配 |
| `is_multi_level` | 否 | 默认 `false` |
| `new_property_name` | 否 | 如果 `is_multi_level` 为 `true` 则必需 |
| `prefix_to_remove` | 否 | 可选，用于多级映射 |

### 更新已预置的映射规则

每次重启时，Keep 读取 `KEEP_MAPPINGS_DIRECTORY` 环境变量并确定哪些映射规则需要添加、移除或更新。

高层预置机制：
1. Keep 读取 `KEEP_MAPPINGS_DIRECTORY` 的值。
2. Keep 列出目录下所有 `.yaml`/`.yml` 文件（其他文件被跳过）。
3. 对于每个清单：按 `name` 查找现有规则。如果找到，更新它并标记 `is_provisioned=True`。如果未找到，创建新的预置规则。
4. 源文件在目录中不再存在的预置规则将被取消预置（删除）。
5. UI 创建的规则（`is_provisioned=False`），其名称未出现在任何清单中，保持不变。

### 采用现有 UI 规则

如果 UI 中已存在与您的清单同名的映射规则，下一次预置运行将**采用**它：`is_provisioned` 翻转为 `True`，规则的 `provisioned_file` 被记录，其内容从清单覆盖。数据库 ID 被保留，因此对该规则的任何外部引用（URL、仪表板）继续有效。

清单模式中不存在的字段（`disabled`、`override`、`condition`）在采用时重置为模型默认值 — 清单是唯一事实来源，因此通过 UI 禁用的规则在被采用时将被重新启用。

### 移除所有预置的映射规则

如果 `KEEP_MAPPINGS_DIRECTORY` 在之前有预置规则的 Keep 实例上被取消设置，所有预置规则将在下次重启时被取消预置（删除）。仅 UI 创建的规则不受影响。

### 每个清单的失败处理

格式错误的清单（无效 YAML、缺少必需字段、验证错误）将被记录并跳过。目录中的其他清单仍然正常处理。每个成功的清单在其自己的事务中提交，因此后续失败不会回滚之前的工作。
