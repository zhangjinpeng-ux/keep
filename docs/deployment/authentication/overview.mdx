Keep 支持各种认证提供商和架构，以适应不同的部署策略和安全需求，从开发环境到生产环境设置。

### 认证提供商

- [**无认证**](/deployment/authentication/no-auth) - 快速设置，适用于测试或内部使用场景。
- [**DB**](/deployment/authentication/db-auth) - 简单的用户名/密码认证。适用于小型团队或开发/预发布环境。用户和哈希密码存储在数据库中。
- [**Auth0**](/deployment/authentication/auth0-auth) - 利用 Auth0 实现可扩展的、基于 auth0 的认证。
- [**Keycloak**](/deployment/authentication/keycloak-auth) - 利用 Keycloak 实现企业级认证方法，如 SSO/SAML/OIDC、具有自定义角色的高级 RBAC、资源级权限以及与用户目录 (LDAP) 的集成。
- [**AzureAD**](/deployment/authentication/azuread-auth) - 利用 Azure AD 实现 SSO/SAML/OIDC 企业级认证。
- [**Okta**](/deployment/authentication/okta-auth) - 利用 Okta 实现 SSO/OIDC 认证。
- [**OneLogin**](/deployment/authentication/onelogin-auth) - 利用 OneLogin 实现 SSO/OIDC 认证。

选择合适的认证策略取决于您的具体使用场景、安全需求和部署环境。您可以阅读更多关于每个认证提供商的信息。

### 如何配置
<Tip>
某些认证提供商需要额外的环境变量。这些将在特定认证提供商页面上详细说明。
</Tip>
Keep 的认证方案通过后端（Keep API）和前端（Keep UI）的环境变量来控制。


|  身份提供商  | 环境变量 | 需要额外变量 |
| ------------------------------------- | -------------------------------------------------------------- | ---------------------------- |
| **无认证** | `AUTH_TYPE=NOAUTH`| 无 |
| **DB** | `AUTH_TYPE=DB`  | `KEEP_JWT_SECRET` |
| **Auth0**                           |  `AUTH_TYPE=AUTH0`  | `AUTH0_DOMAIN`, `AUTH0_CLIENT_ID`, `AUTH0_CLIENT_SECRET` |
| **Keycloak** | `AUTH_TYPE=KEYCLOAK` | `KEYCLOAK_URL`, `KEYCLOAK_REALM`, `KEYCLOAK_CLIENT_ID`, `KEYCLOAK_CLIENT_SECRET` |
| **Oauth2Proxy** | `AUTH_TYPE=OAUTH2PROXY` | `OAUTH2_PROXY_USER_HEADER`, `OAUTH2_PROXY_ROLE_HEADER`, `OAUTH2_PROXY_AUTO_CREATE_USER` |
| **AzureAD** | `AUTH_TYPE=AZUREAD` | 参见 [AzureAD 配置](/deployment/authentication/azuread-auth) |
| **Okta** | `AUTH_TYPE=OKTA` | `OKTA_DOMAIN`, `OKTA_CLIENT_ID`, `OKTA_CLIENT_SECRET` |
| **OneLogin** | `AUTH_TYPE=ONELOGIN` | 参见 [OneLogin 配置](/deployment/authentication/onelogin-auth) |

有关每种认证策略的更多详细信息，包括设置说明和影响，请参阅相应章节。
