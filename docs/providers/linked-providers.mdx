# 链接提供者

在 Keep 中，提供者可以是"已连接"或"已链接"的。理解它们之间的区别对于正确的告警路由和管理非常重要。

</Frame>

## 已连接提供者与已链接提供者

- **已连接提供者（Connected Providers）**：这些是在 Keep 中通过 UI 或 API 显式配置的提供者。它们具有完整的提供者配置和认证详细信息。

- **已链接提供者（Linked Providers）**：这些是向 Keep 发送告警但没有显式连接的提供者。当 Keep 通过 webhook 或推送机制从它们接收告警时，它们会自动出现。

## 链接的工作原理

当 Keep 从未连接的提供者（如 Prometheus 推送告警）接收告警时，它会自动创建一个"已链接"的提供者条目。这使你可以：

- 追踪哪些系统正在发送告警
- 查看 Keep 上次接收告警的时间
- 应用特定于该提供者的去重规则

## 将告警附加到已连接提供者

如果你有一个已连接的提供者，并希望将传入告警关联到它而不是创建链接的提供者，请在 webhook URL 中添加 `provider_id` 查询参数。

例如，对于 Prometheus AlertManager：

```yaml
alertmanager:
  config:
    receivers:
      - name: "keep"
        webhook_configs:
          - url: "https://api.keephq.dev/alerts/event/prometheus?provider_id=your_provider_id"
```

或者对于其他基于 webhook 的集成：

```bash
# Grafana webhook
https://api.keephq.dev/alerts/event/grafana?provider_id=grafana-prod

# Datadog webhook  
https://api.keephq.dev/alerts/event/datadog?provider_id=datadog-main

# 通用 webhook
https://api.keephq.dev/alerts/event/webhook?provider_id=custom-webhook
```

## 最佳实践

1. **对于生产系统**：建议尽可能使用已连接提供者，因为它们提供：

   - 更好的认证和安全性
   - 访问提供者特定功能
   - 更清晰的审计追踪

2. **对于测试/开发**：已链接提供者适用于：

   - 快速原型设计
   - 测试告警流程
   - 临时集成

3. **将已链接转换为已连接**：如果你经常从已链接提供者接收告警，请考虑：
   - 设置正式的提供者连接
   - 使用 `provider_id` 参数将告警附加到已连接提供者

## 限制

已链接提供者：

- 不能用于拉取告警或数据
- 没有认证详细信息
- 不能用于提供者特定的操作
- 可能具有有限的去重能力

如需完整功能，当已链接提供者成为你永久告警基础设施的一部分时，请考虑将它们转换为已连接提供者。
