# 权限

Keep 工作流引擎中的权限定义了**谁可以手动执行工作流**。

它们允许您根据用户角色或特定电子邮件地址限制对工作流的访问，确保只有授权用户才能触发敏感的工作流。

<Note>
目前，权限只能在工作流 YAML 文件中直接编辑。工作流构建器 UI 尚不支持编辑权限。
</Note>

---

## 总体结构

权限在工作流 YAML 文件的顶层使用 `permissions` 字段定义，该字段接受一个角色和/或电子邮件地址的列表。

```yaml
workflow:
  id: sensitive-workflow
  name: Sensitive Workflow
  description: "A workflow with restricted access"
  permissions:
    - admin
    - john.doe@example.com
  steps:
    # workflow steps
```

## 权限如何工作

当工作流定义了权限时：

1. **管理员用户**无论权限列表如何，始终可以运行该工作流
2. **非管理员用户**仅在以下情况下可以运行工作流：
   - 其角色被显式列在权限列表中
   - 或其电子邮件地址被显式列在权限列表中
3. 如果 `permissions` 字段为空或未定义，则任何拥有 `write:workflows` 权限的用户都可以运行该工作流

## 支持的角色类型

Keep 支持以下可在权限列表中使用的角色类型：

- `admin`：拥有完整系统访问权限的管理员用户
- `noc`：具有只读访问权限的网络运维中心用户
- `webhook`：用于 Webhook 集成的 API 访问
- `workflowrunner`：用于通过 API 运行工作流的特殊角色

## 示例

### 仅限管理员用户

```yaml
workflow:
  id: critical-infrastructure-workflow
  name: Critical Infrastructure Workflow
  permissions:
    - admin
  steps:
    # workflow steps
```

### 允许特定用户

```yaml
workflow:
  id: department-specific-workflow
  name: Department Specific Workflow
  permissions:
    - sarah.smith@example.com
    - team.lead@example.com
  steps:
    # workflow steps
```

### 组合角色和个体用户

```yaml
workflow:
  id: mixed-permissions-workflow
  name: Mixed Permissions Workflow
  permissions:
    - admin
    - noc
    - devops.specialist@example.com
  steps:
    # workflow steps
```

## 最佳实践

- 对于对系统有重大影响或触发敏感操作的工作流，使用权限控制
- 对于具有相似职责的用户组，考虑使用基于角色的权限（如 `admin` 或 `noc`）
- 仅在例外情况或需要非常具体的访问控制时才列出个人电子邮件地址
- 作为安全审计的一部分，定期审查工作流权限
- 在内部文档中记录哪些工作流具有受限权限
