---
title: "步骤和动作"
---

步骤（Steps）和动作（Actions）是 Keep 工作流引擎中构建工作流的基础模块。虽然它们共享相似的结构和语法，但**步骤和动作之间的区别主要是语义上的**：

- **步骤**：侧重于从提供者查询数据或触发类似获取的操作（例如，查询数据库、获取日志或检索信息）。
- **动作**：侧重于通知或触发结果，例如发送通知、更新工单或调用外部服务。

步骤和动作共同使工作流既能收集必要的数据，又能对其采取行动。

---

## 总体结构

步骤和动作都使用类似的模式定义：

### 步骤

用于查询或获取数据。

步骤使用每个提供者的 `_query` 方法。

```yaml
steps:
  - name: <step-name>
    provider:
      type: <provider-type>
      config: <provider-config>
      with:
        <provider-specific-parameters>
```

### 动作

用于通知或触发效果。

动作使用每个提供者的 `_notify` 方法。

```yaml

actions:
  - name: <action-name>
    provider:
      type: <provider-type>
      config: <provider-config>
      with:
        <provider-specific-parameters>
```


## 示例


### 从 MySQL 数据库获取数据

```yaml

steps:
  - name: get-user-data
    provider:
      type: mysql
      config: "{{ providers.mysql-prod }}"
      with:
        query: "SELECT * FROM users WHERE id = 1"
        single_row: true
```


### 从 Datadog 获取日志

```yaml
steps:
  - name: get-service-logs
    provider:
      type: datadog
      config: "{{ providers.datadog }}"
      with:
        query: "service:keep and @error"
        timeframe: "1h"
```

### 查询 Kubernetes 运行中的 Pods

```yaml

steps:
  - name: get-pods
    provider:
      type: k8s
      config: "{{ providers.k8s-cluster }}"
      with:
        command_type: get_pods
```

### 发送电子邮件

```yaml
actions:
  - name: send-email
    provider:
      type: email
      config: "{{ providers.email }}"
      with:
        to: "user@example.com"
        subject: "Account Updated"
        body: "Your account details have been updated."
```

### 发送 Slack 消息

```yaml
actions:
  - name: notify-slack
    provider:
      type: slack
      config: "{{ providers.slack-demo }}"
      with:
        message: "Critical alert received!"

```

### 在 ServiceNow 中创建工单

```yaml
actions:
  - name: create-servicenow-ticket
    provider:
      type: servicenow
      config: "{{ providers.servicenow }}"
      with:
        table_name: INCIDENT
        payload:
          short_description: "New incident created by Keep"
          description: "Please investigate the issue."
```

## 组合步骤和动作

工作流通常将步骤（用于查询数据）与动作（用于通知或结果）组合在一起。

以下是几个示例：

### 查询并通知

```yaml
workflow:
  id: query-and-notify
  description: "Query a database and notify via Slack"
  steps:
    - name: get-user-data
      provider:
        type: mysql
        config: "{{ providers.mysql-prod }}"
        with:
          query: "SELECT email FROM users WHERE id = 1"
          single_row: true

  actions:
    - name: send-notification
      provider:
        type: slack
        config: "{{ providers.slack-demo }}"
        with:
          message: "User email: {{ steps.get-user-data.results.email }}"
```

### 告警和事件管理

```yaml
workflow:
  id: alert-management
  description: "Handle alerts and create incidents"
  steps:
    - name: get-alert-details
      provider:
        type: datadog
        config: "{{ providers.datadog }}"
        with:
          query: "service:keep and @alert"
          timeframe: "1h"

  actions:
    - name: create-incident
      provider:
        type: servicenow
        config: "{{ providers.servicenow }}"
        with:
          table_name: INCIDENT
          payload:
            short_description: "Alert from Datadog: {{ steps.get-alert-details.results.alert_name }}"
            description: "Details: {{ steps.get-alert-details.results.alert_description }}"
```

## 错误处理和重试

步骤和动作都支持错误处理，以确保工作流能够从失败中恢复。


```yaml

steps:
  - name: fetch-data
    provider:
      type: http
      with:
        url: "https://api.example.com/data"
    on-failure:
      retry:
        count: 3
        # 每 5 秒重试一次
        interval: 5
```

### 出错时继续执行

默认情况下，如果步骤或动作失败，工作流会立即停止并报告失败。在任意步骤或动作上设置 `continue_on_error: true`，可以吞掉其失败并继续执行剩余的步骤/动作 —— 类似于 GitHub Actions 中的 `continue-on-error`。

```yaml
steps:
  - name: optional-enrichment
    continue_on_error: true
    provider:
      type: http
      config: "{{ providers.some-api }}"
      with:
        url: "https://api.example.com/optional-data"

  - name: always-runs
    provider:
      type: slack
      config: "{{ providers.slack-demo }}"
      with:
        message: "即使 optional-enrichment 失败也会运行"
```

相同的属性也适用于动作：

```yaml
actions:
  - name: best-effort-ticket
    continue_on_error: true
    provider:
      type: servicenow
      config: "{{ providers.servicenow }}"
      with:
        table_name: INCIDENT
        payload:
          short_description: "Automated incident"

  - name: fallback-slack
    provider:
      type: slack
      config: "{{ providers.slack-demo }}"
      with:
        message: "工单创建可能已失败，请检查 ServiceNow"
```

<Note>
当设置 `continue_on_error: true` 时，步骤/动作的失败会被记录为警告，但**不会**将整个工作流执行标记为失败。
</Note>
