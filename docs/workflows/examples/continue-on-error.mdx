---
title: "出错时继续执行"
---

此工作流演示了 `continue_on_error` 的用法，它允许某个步骤或动作失败时不会停止工作流的其余部分 —— 类似于 GitHub Actions 中的 `continue-on-error`。

**用例：** 使用来自外部 API 的可选数据增强告警，然后无论增强是否成功，都始终发送 Slack 通知。

```yaml
workflow:
  id: continue-on-error-example
  description: "Fetch optional enrichment data and always notify via Slack"
  triggers:
    - type: alert
      cel: source.contains("grafana")

  steps:
    - name: fetch-optional-enrichment
      continue_on_error: true
      provider:
        type: http
        config: "{{ providers.internal-api }}"
        with:
          url: "https://api.example.com/enrich/{{ alert.fingerprint }}"
          method: GET

  actions:
    - name: notify-slack
      provider:
        type: slack
        config: "{{ providers.slack-demo }}"
        with:
          message: >
            Alert: {{ alert.name }}
            Enrichment: {{ steps.fetch-optional-enrichment.results | default('unavailable') }}
```

**工作原理：**

1. `fetch-optional-enrichment` 步骤调用外部 API。如果 API 宕机或返回错误，该失败会被记录为警告，执行继续。
2. `notify-slack` 动作始终运行，如果增强结果可用则使用它们，否则优雅地回退。

相同的标志也适用于动作。在以下示例中，会尽力在 ServiceNow 中创建一个工单，并且无论工单创建是否成功，都会发送 Slack 消息：

```yaml
workflow:
  id: best-effort-ticket
  description: "Create a ServiceNow ticket (best-effort) and always notify Slack"
  triggers:
    - type: alert
      cel: severity == "critical"

  actions:
    - name: create-ticket
      continue_on_error: true
      provider:
        type: servicenow
        config: "{{ providers.servicenow }}"
        with:
          table_name: INCIDENT
          payload:
            short_description: "{{ alert.name }}"
            description: "{{ alert.description }}"

    - name: notify-slack
      provider:
        type: slack
        config: "{{ providers.slack-demo }}"
        with:
          message: "Critical alert: {{ alert.name }}. A ServiceNow ticket was attempted."
```

<Note>
当设置 `continue_on_error: true` 时，步骤或动作的失败会被记录为警告，但**不会**将整个工作流执行标记为失败。
</Note>
