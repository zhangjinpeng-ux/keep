# Keep 系统设计文档

> 版本: v0.52.1 | 最后更新: 2026-06-10

---

## 1. 项目概述

**Keep** 是一个开源的 AIOps 告警管理与自动化平台，定位为 "Alerting. For developers, by developers."。它为运维团队提供一站式告警管理能力，覆盖告警的接收、去重、富化、过滤、关联、自动化工作流和可视化的全生命周期。

### 1.1 核心功能

| 能力 | 描述 |
|------|------|
| **统一告警面板** (Single Pane of Glass) | 集中展示所有告警和事件，支持高度可定制的 UI |
| **告警瑞士军刀** | 告警去重 (deduplication)、关联 (correlation)、过滤 (filtering)、富化 (enrichment) |
| **深度集成** | 与监控工具的**双向同步**，支持可定制的工作流 |
| **自动化工作流** | 类似 GitHub Actions 的声明式 YAML 工作流引擎 |
| **AIOps 2.0** | AI 驱动的告警关联与摘要生成 |

### 1.2 技术栈概览

```
┌──────────────────────────────────────────────────────────┐
│                     Keep Platform                        │
├─────────────────────┬────────────────────────────────────┤
│    前端 (keep-ui/)   │        后端 (keep/)                │
│    Next.js 15        │        FastAPI + Uvicorn           │
│    React 19          │        Python >=3.11               │
│    Tailwind CSS      │        SQLModel + SQLAlchemy       │
│    Tremor UI         │        Alembic (DB 迁移)           │
│    CopilotKit (AI)   │        ARQ (异步任务队列)           │
│    Monaco Editor     │        Pusher/Soketi (实时推送)     │
│    React Flow        │        OpenTelemetry (可观测)       │
│    Zustand + SWR     │        Poetry (包管理)              │
└─────────────────────┴────────────────────────────────────┘
```

### 3.4 工作流引擎 (`keep/workflowmanager/`)

#### 3.4.1 工作流 YAML 结构

```yaml
workflow:
  id: sample-workflow
  description: "示例工作流"
  triggers:            # 触发器
    - type: alert      # alert / incident / schedule / manual
      filters:         # 可选的过滤条件
        - key: source
          value: datadog
  steps:               # 步骤（数据查询/富化）
    - name: get-context
      provider:
        type: datadog
        config: "{{ providers.datadog }}"
        with:
          query: "..."
  actions:             # 动作（通知/操作）
    - name: notify-slack
      provider:
        type: slack
        config: "{{ providers.slack }}"
        with:
          message: "Alert: {{ alert.name }}"
```

#### 3.4.2 核心组件

| 组件 | 说明 |
|------|------|
| `WorkflowManager` | 工作流调度管理 — 触发/执行/错误处理 |
| `WorkflowStore` | 工作流持久化存储 — 版本管理 |
| `WorkflowExecution` | 执行记录 — 状态追踪、日志收集 |

#### 3.4.3 执行流程

```
Trigger → Filter → Steps (顺序执行) → Actions (条件执行)
                       ↓                    ↓
               ContextManager         ActionFactory
               (运行时上下文)         (动作路由)
                       ↓                    ↓
               IOHandler             Provider.execute()
               (模板渲染)            (API 调用)
```

### 3.7 事件处理流程

```
外部监控工具 → Webhook → EventSubscriber → process_event_task()
                                               ↓
                                          AlertDeduplicator
                                          (指纹计算 + 去重)
                                               ↓
                                          Enrichment Engine
                                          (映射规则 + AI 富化)
                                               ↓

                                          (告警分组 → Incident)
                                               ↓
                                          Workflow Trigger
                                          (匹配工作流 → 执行)
                                               ↓
                                          [通知 / 工单 / 自定义操作]
```

## 6. 关键数据流

### 6.1 告警处理全流程

```
外部监控工具
    │
    │ Webhook / API Push
    ▼
keep-backend (FastAPI)
    │
    ├── 1. 接收原始告警 → AlertRaw 表
    ├── 2. 指纹计算 (AlertDeduplicator)
    ├── 3. 去重判断
    │       ├── 重复 → 更新 LastAlert，记录去重事件
    │       └── 新告警 → 继续处理
    ├── 4. 富化处理 (Enrichment Engine)
    │       ├── 映射规则 (mapping rules)
    │       ├── 提取规则 (extraction rules)
    │       └── AI 富化 (可选)
    ├── 5. CEL 规则匹配 (Rules Engine)
    │       └── 创建/更新 Incident
    ├── 6. 工作流匹配
    │       └── 执行匹配的工作流 actions
    └── 7. 实时推送 (Pusher/Soketi)
            │
            ▼
      keep-ui (实时更新 UI)
```

### 6.2 工作流执行流程

```
Trigger 触发
    │
    ├── alert trigger → 新告警到达
    ├── incident trigger → 事件状态变更
    ├── schedule trigger → 定时任务
    └── manual trigger → 手动触发
    │
    ▼
Filter 过滤
    │  匹配 trigger.filters 条件
    ▼
Context Resolution (ContextManager)
    │  解析 {{ alert }}, {{ incident }}, {{ steps.xxx }}
    │  解析 {{ providers.xxx }} (密钥注入)
    ▼
Template Rendering (IOHandler)
    │  Mustache/Chevron 模板引擎
    ▼
Step/Action Execution
    │  顺序执行 Steps，条件执行 Actions
    │  foreach 循环支持
    │  if 条件分支支持
    ▼
Result Recording
    │  记录 WorkflowExecution + WorkflowExecutionLog
    │  支持 enrich_alert 回写告警字段
    ▼
实时推送执行状态 (Pusher)
```

### 6.3 实时通信架构

```
keep-backend ──Pusher Event──► Soketi (WebSocket Server)
                                      │
                                      │ WebSocket Connection
                                      ▼
                              keep-ui (Pusher JS Client)
                                      │
                                      ▼
                            SWR Cache Invalidation
                            (自动刷新数据)
```

## 8. 扩展性与定制

### 8.1 Provider 开发

创建新 Provider 只需：

1. 在 `keep/providers/` 下创建目录
2. 继承 `BaseProvider` 实现抽象方法
3. 定义 `PROVIDER_SCOPES` 和 `PROVIDER_METHODS`
4. 系统自动发现并注册

### 8.2 工作流扩展

工作流支持自定义：
- **Steps** — 读取/查询数据作为上下文
- **Actions** — 调用任何 Provider 执行操作
- **Conditions** — assert / threshold / stddev 条件判断
- **Functions** — 内置模板函数（数学/字符串/日期/比较）
- **enrich_alert** — 工作流执行结果回写告警字段

### 8.3 企业版功能 (`ee/`)

| 功能 | 说明 |
|------|------|
| 高级身份管理 | 扩展的 SSO 集成 |
